# wiki optimizer
- 이 문서는 현재 제작중인 wiki optimizer 플러그인의 개발 의도와 컨셉이 작성되어 있는 문서입니다.
- 이 문서는 개발 도중 작성되어 있기 때문에 기존의 구현과 다를 수 있습니다.

# 플러그인 제작 이유
- 이 플러그인의 제작 의도는 claude code, codex cli, gemini cli 등등을 프로그레밍에 활용하면서 굉장히 많은 토큰이 코드베이스 탐색, 관련 코드 파일 필터링, 코드 분석에 소모되는 것을 확인하였습니다.
- 간단한 작업에도 탐색, 필터링, 코드 분석 및 이해의 일련의 ai의 작업 좌정에서 적게는 수만 토큰에서 많게는 백만 이상의 토큰이 사용되는 것을 확인하였습니다.
- 이것을 보고 저는 이렇게 생각하였습니다. 코드베이스를 전부 문서화한 후 색인을 작성해놓으면 ai가 색인을 통해 관련 코드 파일이 무엇인지 빠르게 탐색이 가능하고, 관련 파일만 분석 및 이해하기 때문에 기존보다 사용되는 토큰의 소모량이 줄어들것이라고 생각하였습니다.

# 플러그인의 기능
- 이 플러그인의 핵심 기능은 코드베이스의 코드 파일 문서화, 색인을 통한 코드 문서 탐색,  코드 문서 관리 입니다.

## 문서화
- 문서화 기능은 코드베이스의 코드 파일을 분석하여 하나의 마크다운 파일로 작성합니다.
- 작성된 코드 문서는 해당 코드 파일에 대한 기능 및 역할의 요약이 작성되어 있어 ai가 문서를 확인할 때 문서를 읽고 대상 코드 파일에 대한 사정 정보를 취득하여 코드 파일의 내부 로직과 기능에 대한 이해를 돕습니다.
- 코드 문서는 ai 뿐만 아니라 사람도 읽을 수 있고 관리가 쉽도록 하기 위해 옵시디언 링크 기반 제텔카스텐 기법을 사용하여 색인과 연결을 통해 탐색 및 관리를 생각하고 있습니다.
- 저는 옵시디언 링크 기반 제텔카스텐 기법 도입을 생각하고 있지만 이것이 가장 좋은 방법은 아니라고 생각됩니다. 더 좋은 방법이 있다면 그 방법을 도입할 예정입니다.

## 색인
- 모든 코드 문서는 위키링크를 통헤 색인 문서와 연결되어 있어야 합니다. 색인 문서가 코드베이스 탐색의 시발점이기 때문입니다.
- ai가 작업을 위해 코드베이스 탐색 시 색인 문서를 우선(또는 강제) 확인 합니다.
- 이후 색인 문서를 통해 관련이 있다고 생각되는 문서들을 필터링하여 필터링 된 문서들만을 읽고 분석합니다.
- 이 작업을 통해 작업에 필요하지 않은 코드 파일까지 읽고 분석하는 것을 방지하여 탐색 및 필터링에 사용되는 토큰을 줄일 수 있다고 생각됩니다.

## 관리
- 모든 코드 문서들은 반드시 두 개 이상의 링크를 가지고 있어야 합니다. 최소 두 개를 요구하는 이유는 하나는 색인 문서이고 다른 하나는 문서가 타겟하는 대상 코드 파일입니다. 이 두 개의 링크 이외는 관련 문서 또는 코드 파일 등등 문서를 이해하는데 도움이되는 다른 파일을 연결합니다. 관련 파일 링크는 없어도 되지만 색인 문서와 대상 코드 파일 링크는 반드시 가지고 있어야 합니다.
